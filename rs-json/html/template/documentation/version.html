RS/JSON schemas expected a version attribute. Here is an example:
example.rs.json
{ "version": "0.1.0", "license": "MIT", "charset": "utf8", "schema": {} }
Versioning in RS/JSON has 2 objectives:
  • Giving your the opportunity to propose incremental changes to your schema
  • Let the migration engine aware of changes and take decisions
Incremental changes let your collegues know what kind of changes you operated on your schema, according to semantic versioning principles. This means, if you operated a patch, and your upgrade version from 0.1.0 to 0.1.1, your collegues can figure out it is a fix which has no impact on their data or their app. Migrations engines will take this version into account and operates changes only if the difference between the actual schema and the previous schema is about a minor improvement or a patch. If it is a major improvement, for example from 1.4.3 to 2.0.0, the migration will not operate. With RS/JSON, we want to bring the same advantages we have on code versioning, but for database schemas.
