Bringing the most reliable data is one of the goal of uniqueness. You must have used the same concept if you already dealt with database unique constraints. It is so linked that in fact RS/JSON will turn this directive into an database unique constraint. Let us see a simple example.
book-store.rs.json
{ "version": "0.1.0", "license": "MIT", "charset": "utf8", "schema": { "book": { "uniques": { "unique_book_isbn": ["isbn"] }, "properties": { "name": { "type": "string" }, "isbn": { "type": "isbn" } } } } }
We introduced a uniques attributes which holds, for each key, an array of properties to use for a particular unique constraint. Here, we state that a book cannot have 2 identical ISBN, which is actually natural because ISBN is one of the tool to help identify a book. The uniques attribute can contain constraints referencing multiple properties. Here is another example.
book-store.rs.json
{ "version": "0.1.0", "license": "MIT", "charset": "utf8", "schema": { "author": { "uniques": { "unique_author_name": ["first_name", "last_name"] }, "first_name": { "type": "string" }, "last_name": { "type": "string" } } } }
If you need to give a particular length when indexing a property in a unique constraint, you can do it like so.
book-store.rs.json
{ "version": "0.1.0", "license": "MIT", "charset": "utf8", "schema": { "author": { "uniques": { "unique_author_name": ["first_name:255", "last_name:255"] }, "first_name": { "type": "string" }, "last_name": { "type": "string" } } } }
The instruction "first_name:255" will instruct your storage engine to create an index on the first 255 character of this property. It can be convenient in certain case to manage this index length.
