Guards are validating data coming to your app back-end. This can means a user filling a form or an app using an API for example. This is the advantage of RS/JSON over a relational diagram, you can mix software and database needs, to form a coherent and comprehensive schema. Here is an example of guard validating that a number is greater than 0.
example.rs.json
{ "version": "0.1.0", "license": "MIT", "charset": "utf8", "schema": { "product": { "properties": { "price": { "type": "float", "guards": ["positive"] } } } } }
Guards will perform on the server-side, so if you are creating a database, they are ignored. Guards validate data coming from your back-end, to insert or update one record. So they will not be triggered if you perform an insertion outside an ORM that supports on RS/JSON.
The philosophy of a guard is: validating general feature. Because this is a fuzzy concept, here is a good example of guard.
example.rs.json
{ "version": "0.1.0", "license": "MIT", "charset": "utf8", "schema": { "position": { "properties": { "longitude": { "type": "float", "guards": ["longitude"] } } } } }
This is a well designed guard because it address a general feature, the longitude. Now let us look at a less relevant guard.
example.rs.json
{ "version": "0.1.0", "license": "MIT", "charset": "utf8", "schema": { "position": { "properties": { "longitude": { "type": "float", "guards": ["france-longitude"] } } } } }
This is a bad guard because it contains a business logic. This is not a general purpose guard. Here is how we would use guards to fit our need.
example.rs.json
{ "version": "0.1.0", "license": "MIT", "charset": "utf8", "schema": { "position": { "properties": { "longitude": { "type": "float", "guards": ["longitude", "min:4", "max:8"] } } } } }
This guard is better for two reason. It does not force a business logic inside a guard, so we do not have to dive into the guard code to understand what is happening. It is explicitly stating that the longitude of the France are between 4 and 8 decimal degrees.
